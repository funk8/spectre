      82             : ## Clone the SpECTRE repository
      83             : 
      84             : First, clone the [SpECTRE repository](https://github.com/sxs-collaboration/spectre)
      85             : to a directory of your choice. In the following we will refer to it as
      86             : SPECTRE_ROOT. You may `git clone` from GitHub, in which case SPECTRE_ROOT will
      87             : be `<your_current_directory>/spectre`. That is, inside SPECTRE_ROOT are `docs`,
      88             : `src`, `support`, `tests` etc. You can also download the source and extract them
      89             : to your desired working directory, making sure not to leave out hidden files
      90             : when you `cp` or `mv` the source files.
      91             : 
      92             : ## Using Docker to obtain a SpECTRE environment
      93             : 
      94             : A [Docker](https://www.docker.com/) image is available from
      95             : [DockerHub](https://hub.docker.com/r/sxscollaboration/spectrebuildenv/) and can
      96             : be used to build SpECTRE on a personal machine.
      97             : 
      98             : **Note**: The Docker image or the Singularity image (see below) are
      99             : the recommended ways of using SpECTRE on a personal
     100             : Linux machine. Because of the wide variety of operating systems available today
     101             : it is not possible for us to support all configurations. However, using Spack
     102             : as outlined below is a supported alternative to Docker or Singularity images.
     103             : 
     104             : **Note**: If you have SELinux active
     105             : on your system you must figure out how to enable sharing files with the host
     106             : OS. If you receive errors that you do not have permission to access a shared
     107             : directory it is likely that your system has SELinux enabled. One option is to
     108             : disable SELinux at the expense of reducing the security of your system.
     109             : 
     110             : To build with the Docker image:
     111             : 
     112             : 1. Retrieve the Docker image (you may need `sudo` in front of this command)
     113             :    ```
     114             :    docker pull sxscollaboration/spectrebuildenv:latest
     115             :    ```
     116             : 2. Start the Docker container (you may need `sudo`)
     117             :    ```
     118             :    docker run -v SPECTRE_ROOT:SPECTRE_ROOT --name CONTAINER_NAME \
     119             :               -i -t sxscollaboration/spectrebuildenv:latest /bin/bash
     120             :    ```
     121             :    - `-v SPECTRE_ROOT:SPECTRE_ROOT` binds the directory SPECTRE_ROOT outside the
     122             :    container to SPECTRE_ROOT inside the container. In this way, files in the
     123             :    SPECTRE_ROOT on your host system (outside the container) become accessible
     124             :    within the container through the directory SPECTRE_ROOT inside the
     125             :    container. If you wonder why the same SPECTRE_ROOT needs to be used for
     126             :    both inside and outside the container, which is why `SPECTRE_ROOT` is
     127             :    repeated in the command above with seperated by a colon, please see one of
     128             :    the notes below regarding `-v` flag.
     129             :    - The `--name CONTAINER_NAME` is optional, where CONTAINER_NAME is a name
     130             :    of your choice. If you don't name your container, docker will generate an
     131             :    arbitrary name.
     132             :    - On macOS you can significantly increase the performance of file system
     133             :    operations by appending the flag `:delegated` to `-v`, e.g.
     134             :    `-v SPECTRE_ROOT:SPECTRE_ROOT:delegated` (see
     135             :    https://docs.docker.com/docker-for-mac/osxfs-caching/).
     136             :    - It can be useful to expose a port to the host so you can run servers such
     137             :    as [Jupyter](https://jupyter.org/index.html) for accessing the Python
     138             :    bindings (see \ref spectre_using_python) or a Python web server to view the
     139             :    documentation. To do so, append the `-p` option, e.g. `-p 8000:8000`.
     140             : 
     141             :    You will end up in a bash shell in the docker container,
     142             :    as root (you need to be root).
     143             :    Within the container, the files in SPECTRE_ROOT are available and Charm++ is
     144             :    installed in `/work/charm_6_10_2`. For the following steps, stay inside the
     145             :    docker container as root.
     146             : 3. Proceed with [building SpECTRE](#building-spectre).
     147             : 
     148             : **Notes:**
     149             :   * Everything in your build directory is owned by root, and is
     150             :     accessible only within the container.
     151             :   * You should edit source files in SPECTRE_ROOT in a separate terminal
     152             :     outside the container, and use the container only for compiling and
     153             :     running the code.
     154             :   * If you exit the container (e.g. ctrl-d),
     155             :     your compilation directories are still saved, as are any other changes to
     156             :     the container that you have made.
     157             :     To restart the container, try the following commands
     158             :     (you may need `sudo`):
     159             :     1. `docker ps -a`,
     160             :       to list all containers with their CONTAINER_IDs and CONTAINER_NAMEs,
     161             :     2. `docker start -i CONTAINER_NAME` or `docker start -i CONTAINER_ID`,
     162             :       to restart your container.
     163             :   * When the Docker container gets updated, you can stop it with
     164             :     `docker stop CONTAINER_NAME`, remove it with `docker rm CONTAINER_NAME`
     165             :     and then start at step 2 above to run it again.
     166             :   * You can run more than one shell in the same container, for instance
     167             :     one shell for compiling with gcc and another for compiling
     168             :     with clang.
     169             :     To add a new shell, run `docker exec -it CONTAINER_NAME /bin/bash`
     170             :     (or `docker exec -it CONTAINER_ID /bin/bash`) from
     171             :     a terminal outside the container.
     172             :   * In step 3 above, technically docker allows you to say `-v
     173             :     SPECTRE_ROOT:/my/new/path` to map SPECTRE_ROOT outside the container to any
     174             :     path you want inside the container, but **do not do this**.  Compiling
     175             :     inside the container sets up git hooks in SPECTRE_ROOT that contain
     176             :     hardcoded pathnames to SPECTRE_ROOT *as seen from inside the container*. So
     177             :     if your source paths inside and outside the container are different,
     178             :     commands like `git commit` run *from outside the container* will die with
     179             :     `No such file or directory`.
     180             : 
     181             : ## Using Singularity to obtain a SpECTRE environment
     182             : 
     183             : [Singularity](https://sylabs.io) is a container alternative
     184             : to Docker with better security and nicer integration.
     185             : 
     186             : To build SpECTRE with Singularity you must:
     187             : 
     188             : 1. Build [Singularity](https://sylabs.io) and add it to your
     189             :    `$PATH`
     190             : 2. `cd` to the directory where you want to store the SpECTRE Singularity image,
     191             :    source, and build directories, let's call it WORKDIR. The WORKDIR must be
     192             :    somewhere in your home directory. If this does not work for you, follow the
     193             :    Singularity instructions on setting up additional [bind
     194             :    points](https://sylabs.io/guides/3.7/user-guide/bind_paths_and_mounts.html)
     195             :    (version 3.7. For other versions, see the [docs](https://sylabs.io/docs/)).
     196             :    Once inside the WORKDIR, clone SpECTRE into `WORKDIR/SPECTRE_ROOT`.
     197             : 3. Run `sudo singularity build spectre.img
     198             :    docker://sxscollaboration/spectrebuildenv:latest`.
     199             : 
     200             :    If you get the error message that `makesquashfs` did not have enough space to
     201             :    create the image you need to set a different `SINGULARITY_TMPDIR`. This can
     202             :    be done by running: `sudo SINGULARITY_TMPDIR=/path/to/new/tmp singularity
     203             :    build spectre.img docker://sxscollaboration/spectrebuildenv:latest`. Normally
     204             :    `SINGULARITY_TMPDIR` is `/tmp`, but building the image will temporarily need
     205             :    almost 8GB of space.
     206             : 
     207             :    You can control where Singularity stores the downloaded image files from
     208             :    DockerHub by specifying the `SINGULARITY_CACHEDIR` environment variable. The
     209             :    default is `$HOME/.singularity/`. Note that `$HOME` is `/root` when running
     210             :    using `sudo`.
     211             : 4. To start the container run `singularity shell spectre.img` and you
     212             :    will be dropped into a bash shell.
     213             : 5. Proceed with [building SpECTRE](#building-spectre).
     214             : 
     215             : **Notes:**
     216             : - You should edit source files in SPECTRE_ROOT in a separate terminal
     217             :   outside the container, and use the container only for compiling and running
     218             :   the code.
     219             : - If you don't have the same Python version in your environment outside the
     220             :   container as the version inside the container, this will create problems
     221             :   with git hooks. The Singularity container uses python3.8 by default. Thus, it
     222             :   is up to the user to ensure that they are using the same Python version inside
     223             :   and outside the container. To use a different Python version in the container
     224             :   add `-D Python_EXECUTABLE=/path/to/python` to the cmake command where
     225             :   `/path/to/python` is usually `/usr/bin/pythonX` and `X` is the version you
     226             :   want.
     227             : - Unlike Docker, Singularity does not keep the state between runs. However, it
     228             :   shares the home directory with the host OS so you should do all your work
     229             :   somewhere in your home directory.
     230             : - To run more than one container just do `singularity shell spectre.img` in
     231             :   another terminal.
     232             : - Since the data you modify lives on the host OS there is no need to worry about
     233             :   losing any data, needing to clean up old containers, or sharing data between
     234             :   containers and the host.

     236             : ## Using Spack to set up a SpECTRE environment
     237             : 
     238             : SpECTRE's dependencies can be installed with
     239             : [Spack](https://github.com/spack/spack), a package manager tailored for HPC use.
     240             : [Install Spack](https://spack.readthedocs.io/en/latest/getting_started.html) by
     241             : cloning it into `SPACK_DIR` (a directory of your choice). Then, enable Spack's
     242             : shell support with `source SPACK_DIR/share/spack/setup-env.sh`. Consider adding
     243             : this line to your `.bash_profile`, `.bashrc`, or similar. Refer to [Spack's
     244             : getting started guide](https://spack.readthedocs.io/en/latest/getting_started.html)
     245             : for more information.
     246             : 
     247             : Once you have Spack installed, one way to install the SpECTRE dependencies is
     248             : with a [Spack environment](https://spack.readthedocs.io/en/latest/environments.html):
     249             : 
     250             : \include support/DevEnvironments/spack.yaml
     251             : 
     252             : You can also install the Spack packages listed in the environment file above
     253             : with a plain `spack install` if you prefer.
     254             : 
     255             : **Notes:**
     256             : - Spack allows very flexible configurations and we recommended you read the
     257             :   [documentation](https://spack.readthedocs.io) if you require features such as
     258             :   packages installed with different compilers.
     259             : - For security, it is good practice to make Spack [use the system's
     260             :   OpenSSL](https://spack.readthedocs.io/en/latest/getting_started.html#openssl)
     261             :   rather than allow it to install a new copy.
     262             : - To avoid reinstalling lots of system-provided packages with Spack, use the
     263             :   `spack external find` feature and the `--reuse` flag to `spack concretize` (or
     264             :   `spack install`). You can also install some of the dependencies with your
     265             :   system's package manager in advance, e.g., with `apt` or `brew`. If they are
     266             :   not picked up by `spack external find` automatically, register them with Spack
     267             :   manually. See the [Spack documentation on external
     268             :   packages](https://spack.readthedocs.io/en/latest/build_settings.html#external-packages)
     269             :   for details.
     270             : - Spack works well with a module environment, such as
     271             :   [LMod](https://github.com/TACC/Lmod). See the [Spack documentation on
     272             :   modules](https://spack.readthedocs.io/en/latest/module_file_support.html) for
     273             :   details.
     274             : - On macOS:
     275             :   - Spack's Charm++ 6.10.2 installation is broken, so install `charmpp@7.0.0` or
     276             :     [build Charm++ manually](#building-charm).
     277             :   - Brigand has an issue with AppleClang 13 when compiling tests (see
     278             :     https://github.com/edouarda/brigand/issues/274). Since it is header-only,
     279             :     you can simply clone the [Brigand repository](https://github.com/edouarda/brigand)

     282             : ## Building Charm++ {#building-charm}
     283             : 
     284             : If you are not using a container, haven't installed Charm++ with Spack, or want
     285             : to install Charm++ manually for other reasons, follow the installation
     286             : instructions in the [Charm++ repository](https://github.com/UIUC-PPL/charm)
     287             : and in their [documentation](https://charm.readthedocs.io/en/latest/quickstart.html#installing-charm).
     288             : Here are a few notes:
     289             : 
     290             : - Once you cloned the [Charm++ repository](https://github.com/UIUC-PPL/charm),
     291             :   run `git checkout v6.10.2` to switch to a supported, stable release of
     292             :   Charm++.
     293             : - Choose the `LIBS` target to compile. This is needed so that we can support the
     294             :   more sophisticated load balancers in SpECTRE executables.
     295             : - On a personal machine the correct target architecture is likely
     296             :   `multicore-linux-x86_64`, or `multicore-darwin-x86_64` on macOS. On an HPC
     297             :   system the correct Charm++ target architecture depends on the machine's
     298             :   inter-node communication architecture. It might take some experimenting to
     299             :   figure out which Charm++ configuration provides the best performance.
     300             : - When compiling Charm++ you can specify the compiler using, for example,
     301             :   ```
     302             :   ./build LIBS ARCH clang
     303             :   ```

     305             : ## Building SpECTRE {#building-spectre}
     306             : 
     307             : Once you have set up your development environment you can compile SpECTRE.
     308             : Follow these steps:
     309             : 
     310             : 1. Create a build directory where you would like to compile SpECTRE. In the
     311             :    Docker container you could create, e.g., `/work/spectre-build`. It can be
     312             :    useful to add a descriptive label to the name of the build directory since
     313             :    you may create more later, e.g., `build-clang-Debug`. Then, `cd` into the
     314             :    build directory.
     315             : 2. Determine the location of your Charm++ installation. In the Docker container
     316             :    it is `/work/charm_6_10_2/multicore-linux-x86_64-gcc` for GCC builds and
     317             :    `/work/charm_6_10_2/multicore-linux-x86_64-clang` for clang builds. For Spack
     318             :    installations you can determine it with
     319             :    `spack location --install-dir charmpp`. We refer to the install directory as
     320             :    `CHARM_ROOT` below.
     321             : 3. In your new SpECTRE build directory, configure the build with CMake:
     322             :    ```
     323             :    cmake -D CHARM_ROOT=$CHARM_ROOT SPECTRE_ROOT
     324             :    ```
     325             :    Add options to the `cmake` command to to configure the build, select
     326             :    compilers, etc. For instance, to build with clang you may run:
     327             :    ```
     328             :    cmake -D CMAKE_CXX_COMPILER=clang++ \
     329             :          -D CMAKE_C_COMPILER=clang \
     330             :          -D CMAKE_Fortran_COMPILER=gfortran \
     331             :          -D CHARM_ROOT=$CHARM_ROOT \
     332             :          SPECTRE_ROOT
     333             :    ```
     334             :    See \ref common_cmake_flags for documentation on possible configuration
     335             :    options.
     336             : 4. When cmake configuration is done, you are ready to build target executables.
     337             :    - You can see the list of available targets by running `make list` (or `ninja
     338             :      list` if you are using the Ninja generator) or by using tab completion.
     339             :      Compile targets with `make -jN TARGET` (or `ninja -jN TARGET`), where `N`
     340             :      is the number of cores to build on in parallel (e.g. `-j4`). Note that the
     341             :      Ninja generator allows you to compile individual source files too.
     342             :    - Compile the `unit-tests` target and run `ctest -L unit` to run unit tests.
     343             :      Compile `test-executables` and run `ctest` to run all tests, including
     344             :      executables. To compile `test-executables` you may have to reduce the
     345             :      number of cores you build on in parallel to avoid running out of memory.
     346             :    - To compile the Python bindings, add the option
     347             :      `-D BUILD_PYTHON_BINDINGS=ON` to the `cmake` command and compile the
     348             :      `all-pybindings` target (see \ref spectre_using_python).
