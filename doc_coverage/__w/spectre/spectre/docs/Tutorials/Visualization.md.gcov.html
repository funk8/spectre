       5             : # Running and Visualizing {#tutorial_visualization}
       6             : 
       7             : ### Building an Executable from an Existing Source File
       8             : 
       9             : SpECTRE source files for evolution executables are located in
      10             : `src/Evolution/Executables`. Executables can be compiled by running the command
      11             : `make EXECUTABLE_NAME` where `EXECUTABLE_NAME` is the name of the executable
      12             : as defined in the `CMakeLists.txt` file located in the same directory as the
      13             : source file. For example, to compile the executable that evolves a scalar wave
      14             : using a three-dimensional domain, one runs the command:
      15             : `make EvolvePlaneWave3D`, which then results in an executable of the same name
      16             : in the `bin` directory of the user's build directory.
      17             : 
      18             : ### Running an Evolution Executable
      19             : 
      20             : Each SpECTRE executable reads a user-provided YAML file that specifies the
      21             : runtime settings of the evolution. This is where options such as the Domain,
      22             : AnalyticSolution, TimeStepper, etc are supplied. Example input files are kept
      23             : in `tests/InputFiles`, and are written to provide a functional base input file
      24             : which the user can then modify as desired. Copy the executable and YAML file
      25             : to a directory of your choice. The YAML file is then passed as an argument to
      26             : the executable using the flag `--input-file`. For example, for a scalar wave
      27             : evolution, run the command:
      28             : `./EvolvePlaneWave3D --input-file Input3DPeriodic.yaml`.
      29             : By default, the example input files do not produce any output. This can be
      30             : changed by modifying the options passed to `EventsAndTriggers` or
      31             : `EventsAndDenseTriggers`:
      32             : 
      33             : \snippet PlaneWave1DObserveExample.yaml observe_event_trigger
      34             : 
      35             : This will observe the norms of the errors in the system at times
      36             : \f$0.0\f$ and \f$1.0\f$ and the field data at the start of every 50th
      37             : slab.  A successful observation will result in the creation of H5
      38             : files whose names can be specified in the YAML file under the options
      39             : `VolumeFileName` and `ReductionFileName`. One volume data file will be
      40             : produced from each Charm++ node that is used to run the
      41             : executable. Each volume data file will have its corresponding node
      42             : number appended to its file name.  Visualization of the volume data
      43             : will be described in the next section.
      44             : 
      45             : ### 3D Data Volume Data In ParaView
      46             : 
      47             : A SpECTRE executable with observers produces volume and/or reduced data h5
      48             : files. An XDMF file must be created from the volume data in order to do
      49             : visualization using ParaView. To this end we provide the python executable
      50             : `GenerateXdmf.py` in the `Visualization/Python` directory. `GenerateXdmf.py`
      51             : takes three required arguments which are passed to `--file-prefix`,
      52             : `--subfile-name`, and `--output`. The argument passed to `--file-prefix` is the
      53             : name of the H5 volume data, leaving out the node number and extension. The
      54             : argument passed to `--subfile-name` is the name of the subfile inside the H5
      55             : volume data file where data is stored. In the above example, `--subfile-name`
      56             : would be `VolumePsiPiPhiEvery50Slabs`. The argument passed to `--output` is the
      57             : desired .xmf file name, also without filename extension. Use `--help` to see a
      58             : further description of possible arguments.
      59             : 
      60             : Open the .xmf file in ParaView and select the `Xdmf Reader`, *not* the version 3
      61             : readers. On the left hand side of the main ParaView window is a section named
      62             : `Properties`, here you must click the highlighted `Apply` button. ParaView will
      63             : now render your volume data. If you only wish to visualize a few datasets out of
      64             : a large set, we recommended unchecking the boxes for the datasets you wish to
      65             : ignore under `Point Arrays` before clicking `Apply`. Also, don't forget to check
      66             : which dataset is being used for the color under `Coloring`. The default won't
      67             : necessarily be the dataset you want.
      68             : 
      69             : ### Helpful ParaView Filters
      70             : 
      71             : Here we describe the usage of filters we've found to better visualize our data.
      72             : Feel free to contribute to this section!
      73             : 
      74             : #### Removing Mesh Imprinting
      75             : You may notice what appears to be mesh imprinting on the data. The imprinting
      76             : effect can be removed by applying the `Tetrahedralize` filter. To apply the
      77             : filter select the `Filters` menu item, then `Alphabetical` and finally
      78             : `Tetrahedralize`.
      79             : 
      80             : #### Creating Derived Volume Data
      81             : New volume data can be created from the existing volume data using the
      82             : `Calculator` filter. In the `Calculator`'s text box, input a numerical
      83             : expression in terms of existing datasets evaluating to the desired
      84             : quantity. For example, a vector-valued velocity dataset can be created
      85             : from three scalar velocity component datasets using the expression
      86             : `velocity_x * iHat + velocity_y * jHat + velocity_z * kHat` and hitting
      87             : the `Apply` button. By default, this will create a new dataset `Result`.
      88             : The name of the new dataset can be changed by changing the name provided
      89             : to `Result Array Name` above the `Calculator` text box.
      90             : 
      91             : #### Visualizing Vector Fields Derived From Scalar Fields
      92             : Use the `Calculator` filter described above in "Creating Derived Volume Data"
      93             : to create a new vector-valued dataset. Once this is created, use the `Glyph`
      94             : filter and set the `Active Attributes` to the vector you wish to visualize.
      95             : Make sure that `Scale Mode` is set to `vector`.
      96             : 
      97             : ### Reproducibility of Results
      98             : 
      99             : Being able to reproduce the results of simulations is important for scientific
     100             : integrity and accuracy. To this end, SpECTRE encodes the source tree,
     101             : environment variables, and a record of build information (e.g., the third-party
     102             : library versions) into the executables. From most executables these can be
     103             : retrieved using the flags `--dump-source-as filename_without_extension`,
     104             : `--dump-paths`, `--dump-build-info`, and `--dump-environment`. The flag
     105             : `--dump-only` will cause the program to exit after the information has been
     106             : dumped. You can use the `--help` flag to get more details on what flags are
     107             : available. The flag `--dump-paths` will print to screen the environment
     108             : variables `PATH`, `CPATH`, `LD_LIBRARY_PATH`, `LIBRARY_PATH`, and
     109             : `CMAKE_PREFIX_PATH` at link time. The flag `--dump-build-info` will print to
     110             : screen the contents of the `BuildInfo.txt` file generated by CMake. The flag
     111             : `--dump-environment` will print to screen the output of `printenv` at link
     112             : time. Finally, the flag `--dump-source-as filename_without_extension` will dump
     113             : the archive `filename_without_extension.tar.gz`, which contains the entire
     114             : source tree at link time. The combined above information should be enough to
     115             : reproduce simulations results to roundoff error as long as the third party
     116             : libraries used to build the executable have not changed on the system the
     117             : simulation was run on.
     118             : 
     119             : Often only data from the simulation is retained, not the entire executable. In
     120             : order to make simulations reproducible from only the HDF5 output, any HDF5 files
     121             : written with SpECTRE's HDF5 wrappers contain the info about the run, output of
     122             : `printenv` at link time, the contents of `BuildInfo.txt`, and an archive
     123             : of the source tree. These are inside the `header.hdr` group with names
     124             : `SimulationInfo`, `EnvironmentVariablesAtCompilation`, `BuildInfo`, and
     125             : `src.tar.gz`. The source archive can be extracted by running the command
     126             : ```
     127             : h5dump -d /src.tar.gz -b LE -o src.tar.gz /path/to/hdf5/file.h5
     128             : ```
     129             : You will need to make sure the argument to `-b` is the same endianness as your
     130             : computer. In the above example little endian is used.
